## 第一部分：ReAct理论讲解（20分钟）

---

### Slide 1: 什么是ReAct？

**ReAct = Reasoning（推理）+ Acting（行动）**

```
核心思想：
让AI像人类一样思考和行动

Thought（思考）→ Action（行动）→ Observation（观察）→ 循环
```

**讲师要点**：
- ReAct来自论文《ReAct: Synergistic Reasoning and Acting in Language Models》（2022）
- 解决了传统LLM缺乏推理和行动能力的问题
- 让AI能够"想清楚再做"

---

### Slide 2: 为什么需要ReAct？

**传统LLM的问题**：

| 问题 | 表现 | 后果 |
|------|------|------|
| 缺乏推理 | 直接给出答案，过程不可见 | 错误难定位 |
| 无法行动 | 只能生成文本 | 无法调用外部工具 |
| 无记忆 | 不记录中间步骤 | 无法处理复杂任务 |
| 缺乏反思 | 一条路走到黑 | 错误无法纠正 |

**讲师要点**：
- 举例子：如果问"四川电网2024年2月用电量"，传统LLM会直接编造答案
- ReAct会先思考、再行动、观察结果、继续思考

---

### Slide 3: ReAct循环流程

```
┌─────────────────────────────────────┐
│      ReAct 循环流程                  │
├─────────────────────────────────────┤
│                                     │
│  用户提问                            │
│     ↓                               │
│  ┌──────────┐                       │
│  │ Thought  │ ← 思考：我需要做什么？  │
│  └──────────┘                       │
│     ↓                               │
│  ┌──────────┐                       │
│  │ Action   │ ← 行动：执行具体操作   │
│  └──────────┘                       │
│     ↓                               │
│  ┌──────────┐                       │
│  │Observation│ ← 观察：得到什么结果？│
│  └──────────┘                       │
│     ↓                               │
│  [是否完成？] ─否→ 返回Thought      │
│     ↓ 是                            │
│  最终答案                           │
│                                     │
└─────────────────────────────────────┘
```

**讲师要点**：
- 强调这是一个循环过程
- 每一步都有明确的输入输出
- 过程透明、可追溯

---

### Slide 4: ReAct实例演示

**案例：查询四川电网2024年2月用电量**

```
Thought（思考）:
用户想查询四川电网的用电量数据，需要：
- 时间范围：2024年2月
- 查询对象：四川电网
- 查询指标：用电量

Action（行动）:
调用数据查询API，参数：{
  "region": "四川电网",
  "time_range": "2024-02",
  "metric": "用电量"
}

Observation（观察）:
查询结果：
- 四川省2024年2月用电量：256.8亿千瓦时
- 同比增长：5.2%
- 数据来源：营销系统

Thought（再思考）:
已获取到用电量数据，需要生成自然语言回复

Action（最终行动）:
生成回答："四川省电网2024年2月用电量为256.8亿千瓦时，同比增长5.2%。"

Final Answer: ...
```

**讲师要点**：
- 用真实案例讲解
- 强调每一步的作用
- 对比直接给出答案的优势

---

### Slide 5: ReAct vs 传统方法

| 维度 | 传统Prompt | ReAct |
|------|-----------|-------|
| **可解释性** | 黑盒，过程不可见 | 透明，每步可见 |
| **工具调用** | 无法调用 | 主动调用 |
| **复杂任务** | 难以处理 | 分解执行 |
| **错误定位** | 难以定位 | 每步可查 |
| **适用场景** | 简单问答 | 复杂多步骤任务 |

**讲师要点**：
- ReAct不是万能的，简单任务不需要
- 适合需要推理和多步骤的任务

## 第二部分：灵知平台实现方案（25分钟）

---

### Slide 6: 灵知平台ReAct实现思路

**Plan-based ReAct**

```
传统ReAct：动态生成Action
  ↓
Plan-based ReAct：预先规划Plan
  ↓
在灵知平台上实现
```

**核心思想**：
1. 用户提问 → 生成执行计划（Action List）
2. 循环遍历Action List
3. 根据Action类型激活不同工具路径
4. 执行并验证结果
5. 判断是否完成，或补充Plan

**讲师要点**：
- 为什么选择Plan-based：适合灵知平台的模块化特性
- 不是真正的"动态思考"，而是"预先规划+动态执行"

---

### Slide 7: 三大执行阶段

```
Phase 1: Plan Generation
  ├─ 分析用户问题
  ├─ 生成Action List
  └─ 初始化Memory

Phase 2: Execution Loop
  ├─ 遍历Action List
  ├─ 激活对应工具路径
  ├─ 执行工具并验证
  └─ 更新Memory

Phase 3: Completion Check
  ├─ 判断任务完成度
  ├─ 生成最终答案
  └─ 或补充Action List
```

**讲师要点**：
- 每个阶段都有明确的输入输出
- Phase 2是核心，需要详细讲解

---

### Slide 8: Phase 1 - Plan Generation

**模块流程**：

```
用户提问 
  → 智能对话（生成Plan）
  → 信息提取（提取Action List）
  → 代码块（初始化Memory）
  → Memory写入
```

**关键提示词**：

```
你是一个任务规划专家。

可用工具：
1. knowledge_search: 知识库搜索
2. api_call: API调用
3. db_query: 数据库查询
4. doc_gen: 文档生成
5. code_exec: 代码执行
6. external_tool: 外部工具调用

用户问题：{{user_question}}

请生成详细的执行计划...
```

**讲师要点**：
- 强调工具清单的重要性
- 提示词设计是关键
- Plan质量直接影响执行效果

---

### Slide 9: Phase 2 - Execution Loop（核心）

**循环体结构**：

```
循环模块（For Each Action）
  ├─ 智能对话（读取Memory）
  ├─ 信息提取（提取tool_type）
  ├─ 代码块（路径激活）
  │
  ├─ [知识库搜索路径] → 验证
  ├─ [API调用路径]    → 验证
  ├─ [数据库查询路径] → 验证
  ├─ [文档生成路径]   → 验证
  ├─ [代码执行路径]   → 验证
  └─ [外部工具路径]   → 验证
      ↓
  循环终点 → 返回起点
```

**讲师要点**：
- 6个工具路径并行
- 每个路径结构类似：执行→验证→更新Memory
- 重点讲解路径激活机制

---

### Slide 10: 路径激活机制

**代码块实现**：

```javascript
function userFunction(input) {
  const toolType = input.tool_type;
  
  return {
    "is_knowledge_search": toolType === "knowledge_search",
    "is_api_call": toolType === "api_call",
    "is_db_query": toolType === "db_query",
    "is_doc_gen": toolType === "doc_gen",
    "is_code_exec": toolType === "code_exec",
    "is_external_tool": toolType === "external_tool"
  };
}
```

**激活条件**：
- 知识库搜索路径：`is_knowledge_search == True`
- API调用路径：`is_api_call == True`
- 其他路径类似

**讲师要点**：
- 这是灵知平台的特色功能
- 通过布尔值控制路径激活
- 比传统的if-else更灵活

---

### Slide 11: 重试与失败处理

**重试机制**：

```
验证结果 
  → 成功：更新Memory，继续下一Action
  → 失败：重试次数<3？
       ├─ 是：更新Memory，重试当前Action
       └─ 否：标记失败，继续下一Action
```

**重试实现**：

```javascript
// 跳出循环，用新数组重新启动
if (retryCount < 3) {
  // 生成新数组：当前Action + 后续Action
  const newActionList = [...remainingActions];
  
  return {
    "exit_loop": true,
    "new_action_list": newActionList
  };
}
```

**讲师要点**：
- 灵知平台的循环模块可以从内部跳出
- 用新数组重新启动循环
- 需要设计好Memory和上下文

---

### Slide 12: Phase 3 - Completion Check

**完成判断逻辑**：

```
退出循环 
  → 智能对话（判断完成）
  → 条件判断
       ├─ 完成 → 生成最终答案 → 输出用户
       └─ 未完成 → 补充Action List → 返回Phase 2
```

**关键提示词**：

```
Memory内容：{{memory}}

用户原始问题：{{memory.context.user_question}}

已执行的Action：{{memory.results}}

请判断：
1. 任务是否完成？
2. 完成度是多少？
3. 还缺少什么？
4. 是否需要补充Action？
```

**讲师要点**：
- 这是ReAct的"反思"能力
- 可以根据执行结果调整策略
- 实现了动态优化

## 第三部分：关键技术点（10分钟）

---

### Slide 13: Memory管理

**Memory结构设计**：

```json
{
  "plan": {
    "task_understanding": "任务理解",
    "actions": [...]
  },
  "current_index": 0,
  "retry_counts": {},
  "results": {},
  "context": {
    "user_question": "",
    "iteration": 0,
    "summary": ""
  }
}
```

**Memory读写方式**：

- **写入**：代码块输出 → 连线 → Memory模块（自动写入变量）
- **读取**：在提示词中使用 `{{memory}}` 引用

**讲师要点**：
- Memory是全局状态管理
- 所有模块共享Memory
- 注意Memory大小控制

---

### Slide 14: 上下文控制

**上下文传递机制**：

```
方案1：通过Memory传递
  每次Action执行后，更新memory.context

方案2：通过智能对话的聊天上下文
  配置聊天上下文为3-5轮

方案3：专门的信息加工模块
  在工具执行后，生成上下文摘要
```

**上下文使用示例**：

```
提示词：

历史执行情况：
{{memory.action_results}}

当前任务上下文：
{{memory.context}}

请基于以上上下文，执行当前Action...
```

**讲师要点**：
- 上下文是ReAct的"记忆"
- 确保每一步都能了解之前发生了什么
- 避免重复操作

---

### Slide 15: 关键提示词设计

**提示词设计原则**：

1. **结构化输出**
   - 强制输出JSON格式
   - 便于后续处理

2. **明确约束**
   - 限制Action数量
   - 明确参数格式

3. **提供上下文**
   - 引用Memory内容
   - 说明当前状态

4. **错误处理**
   - 要求输出错误信息
   - 提供降级方案

**示例模板**：

```
角色：任务规划专家

约束条件：
- Action数量控制在10个以内
- 每个Action必须有明确的成功标准
- 参数必须具体可执行

输出格式（JSON）：
{...}

只输出JSON。
```

**讲师要点**：
- 提示词是AI的"指令"
- 好的提示词=成功的一半
- 需要反复优化

### Slide 17: 常见问题与解决方案

| 问题 | 解决方案 |
|------|----------|
| Memory过大 | 定期清理，只保留关键结果 |
| 死循环 | 设置最大循环次数（50次） |
| 重试无限 | 严格限制重试次数（3次） |
| API超时 | 设置超时时间，添加重试 |
| Plan质量差 | 优化提示词，添加验证步骤 |

**讲师要点**：
- 提前预防常见问题
- 做好错误处理
- 记录详细日志便于调试

### Slide 18: 实施注意事项

**关键注意事项**：

1. ✅ **Memory大小控制**
   - 避免存储过大数据
   - 定期清理

2. ✅ **代码块执行时间**
   - 保持代码简洁
   - 必要时分拆逻辑

3. ✅ **循环次数限制**
   - 设置最大循环次数
   - 防止死循环

4. ✅ **错误处理**
   - 每个模块考虑异常
   - 提供降级方案

5. ✅ **性能优化**
   - 减少不必要的对话
   - 优化提示词长度

1. ✅ **适配灵知平台**：充分利用模块化特性

## 附录：讲师答疑要点

### 常见学员问题预判

**Q1：为什么要用Plan-based ReAct，而不是真正的动态ReAct？**

A: 灵知平台的特性决定了Plan-based更适合：
- 平台的循环模块需要预先定义数组
- 动态生成Action需要复杂的代码逻辑
- Plan-based在可控性和可调试性上更好

**Q2：6个工具路径是否都要实现？**

A: 不一定，根据项目需求：
- 智能问数：主要用api_call、db_query
- 智能写作：主要用knowledge_search、doc_gen
- 工单处理：主要用knowledge_search、code_exec

**Q3：Memory会不会太大导致性能问题？**

A: 需要控制：
- 只存储关键结果，不存储完整数据
- 定期清理不需要的历史记录
- 使用摘要代替完整内容

**Q4：如果重试3次都失败怎么办？**

A: 设计降级方案：
- 标记该Action失败，继续后续Action
- 在完成判断阶段，根据失败情况调整Plan
- 最终答案中说明哪些部分未能完成

**Q5：提示词如何优化？**

A: 多次迭代：
- 从简单开始，逐步完善
- 根据实际执行结果调整
- 参考详细文档中的提示词模板
