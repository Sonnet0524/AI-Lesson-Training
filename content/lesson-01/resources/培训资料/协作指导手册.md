# 小组协作指导手册

**适用对象**：国网四川电力AI种子团队培训学员  
**最后更新**：2024年3月3日

---

## 一、协作概述

### 1.1 培训分组

**分组方案**：
- 总人数：约80人
- 分组数量：12-14组
- 每组人数：6-7人

**分组原则**：
- 业务背景多样性
- 技术能力均衡性
- 地域部门分布

### 1.2 学习模式

**全员学习 + 角色分工**：
```
学习阶段：
✅ 所有学员学习全部课程内容
✅ 掌握Agent开发全流程
✅ 掌握前端开发基础

执行阶段：
🔄 根据角色分工协作
🔄 共同完成项目交付
🔄 互相学习，共同进步
```

**核心理念**：
- 🎯 目标：每个学员都能独立完成端到端开发
- 🤝 协作：通过分工提高效率和代码质量
- 📚 学习：在协作中互相学习，共同成长

## 二、角色职责

### 2.1 产品经理

**主要职责**：
```
需求管理：
- 理解和细化业务需求
- 编写需求文档和用户故事
- 确定功能优先级

项目协调：
- 主持每日站会
- 跟踪项目进度
- 协调组内资源
- 对外沟通汇报

质量把控：
- 验收功能实现
- 收集用户反馈
- 提出优化建议

**工作时间分配**：
```
Day 1-2：需求梳理、项目规划（50%）

## 三、协作流程

### 3.1 每日站会流程

**时间**：每天上午9:00，时长15分钟

**站会流程**：
```
1. 回顾昨日（3分钟/人）
   - 完成了什么？
   - 遇到什么问题？
   - 学到了什么？

2. 计划今日（3分钟/人）
   - 今天要做什么？
   - 需要什么帮助？
   - 有什么风险？

3. 问题讨论（5分钟）
   - 需要组内协调的问题
   - 需要讲师支持的问题
   - 其他共同事项
```

**站会规则**：
- ⏰ 准时开始，不等待迟到者
- 📢 每人发言不超过3分钟
- 🎯 聚焦重点，不展开详细讨论
- 📝 记录问题和行动项

**站会记录模板**：
```
日期：2024-03-XX
参会人员：[姓名列表]

昨日完成：
- 张三：完成Agent流程设计
- 李四：完成环境配置
- 王五：学习React基础

今日计划：
- 张三：开发智能对话模块
- 李四：对接知识库
- 王五：创建前端项目

问题与风险：
- 问题1：知识库文档不足
  负责人：李四
  解决方案：联系助教获取
  
- 问题2：前端样式问题
  负责人：王五
  解决方案：组内讨论或请教讲师
```

**步骤1：需求拆解**

我想要通过自然语言查询用电数据
以便快速获取所需信息，提高工作效率

```
技术负责人协助：

2. 评估任务难度和工作量
3. 根据成员能力和兴趣分配

├────────────────────────────┤

│ □ 返回格式正确             │
│ □ 异常情况处理完善         │
└────────────────────────────┘
```

**步骤3：进度跟踪**
```
产品经理负责：
1. 维护任务看板（Kanban）
2. 每日更新任务状态
3. 识别阻塞和风险

任务状态：
📋 待办 (To Do)
🔄 进行中 (In Progress)
👀 审查中 (Review)

```

## 四、Git协作规范

### 4.1 分支管理策略

**主分支**：
```
master/main
├── 稳定的、可发布的代码
├── 只通过merge request合并
└── 禁止直接提交
```

**开发分支**：
```
develop
├── 开发主线
├── 整合各功能分支
└── 定期合并到master
```

**功能分支**：
```
feature-xxx
├── 新功能开发
├── 从develop分支创建
└── 完成后合并回develop
```

**修复分支**：
```
bugfix-xxx
├── Bug修复
├── 从develop分支创建
└── 修复后合并回develop
```

**分支命名规范**：
```
feature-[功能名]     # 如：feature-smart-query
bugfix-[问题描述]    # 如：bugfix-api-error
hotfix-[紧急问题]    # 如：hotfix-login-fail
release-[版本号]     # 如：release-v1.0
```

### 4.2 提交信息规范

**格式**：
```
<类型>: <简短描述>

<详细描述>（可选）

<页脚>（可选）
```

**类型说明**：
```
feat:     新功能
fix:      Bug修复
docs:     文档更新
style:    代码格式（不影响功能）
refactor: 重构（不是新功能也不是Bug修复）
test:     测试相关
chore:    构建/工具相关
```

**示例**：
```bash
# 好的提交信息
git commit -m "feat: 添加智能问数Agent基础流程"
git commit -m "fix: 修复知识库搜索返回格式错误"
git commit -m "docs: 更新API文档，添加调用示例"

# 不好的提交信息
git commit -m "更新"
git commit -m "fix bug"
git commit -m "修改了一些东西"
```

### 4.3 代码审查流程

**Pull Request流程**：
```
1. 创建功能分支
   git checkout -b feature-smart-query

2. 开发并提交
   git add .
   git commit -m "feat: 实现智能问数功能"
   git push origin feature-smart-query

3. 在Gitee创建Pull Request
   - 填写PR标题和描述
   - 关联相关Issue
   - 指定审查人

4. 代码审查
   - 审查人检查代码
   - 提出修改建议
   - 讨论和改进

5. 合并代码
   - 审查通过
   - 合并到develop或master
   - 删除功能分支
```

**审查要点**：
```
代码质量：
✅ 代码可读性
✅ 命名规范
✅ 注释完整性
✅ 无冗余代码

功能实现：
✅ 功能完整性
✅ 逻辑正确性
✅ 异常处理

测试验证：
✅ 基本功能测试
✅ 边界情况测试
✅ 兼容性考虑
```

### 4.4 冲突解决方法

**预防冲突**：
```
1. 及时同步代码
   git pull origin develop

2. 小步提交
   - 每个功能点独立提交
   - 避免大范围修改

3. 及时沟通
   - 修改公共文件前通知组员
   - 避免同时修改同一文件
```

**解决冲突步骤**：
```
1. 拉取最新代码
   git pull origin develop

2. Git提示冲突
   CONFLICT (content): Merge conflict in file.js

3. 打开冲突文件
   <<<<<<< HEAD
   你的修改
   =======
   他人的修改
   >>>>>>> branch-name

4. 手动解决冲突
   - 保留需要的代码
   - 删除冲突标记
   - 确保代码逻辑正确

5. 提交解决结果
   git add 冲突文件
   git commit -m "chore: 解决合并冲突"
   git push origin your-branch
```

**常见冲突场景**：
```
场景1：多人修改同一文件的同一位置
解决：沟通协商，确定最终方案

场景2：文件重命名或移动
解决：统一文件结构，及时通知组员

场景3：依赖版本冲突
解决：统一依赖版本，更新package.json
```

## 五、前后端协作

### 5.1 API接口对接流程

**步骤1：接口文档准备**
```
Agent开发者负责：
1. 明确接口功能
2. 定义接口规范：
   - 请求方法（GET/POST）
   - 请求路径
   - 请求参数
   - 返回格式
   - 错误码

示例文档：
## 智能问数接口

**接口路径**: POST /api/smart-query
**功能描述**: 自然语言查询电力数据

**请求参数**:
```json
{
  "question": "查询四川省2024年2月的用电量",
  "user_id": "user001"
}
```

**返回示例**:
```json
{
  "success": true,
  "data": {
    "answer": "四川省2024年2月用电量为XXX亿千瓦时",
    "chart_data": {
      "labels": ["2024-02"],
      "values": [XXX]
    }
  },
  "error": null
}
```

**错误码**:
- 400: 参数错误
- 500: 服务器错误
```

**步骤2：接口开发**
```
Agent开发者：
1. 在灵知平台配置Agent
2. 设置API发布
3. 测试接口可用性
4. 提供测试数据

前端开发者：
1. 阅读接口文档
2. 理解数据格式
3. 准备Mock数据（如果接口未就绪）
```

**步骤3：接口对接**
```
前端开发者：
1. 封装API调用函数
2. 处理请求和响应
3. 异常情况处理
4. 联调测试

代码示例：
// api.js
export async function smartQuery(question) {
  try {
    const response = await fetch('/api/smart-query', {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
      },
      body: JSON.stringify({ question }),
    });
    
    const data = await response.json();
    
    if (!data.success) {
      throw new Error(data.error || '查询失败');
    }
    
    return data.data;
  } catch (error) {
    console.error('智能问数错误:', error);
    throw error;
  }
}
```

### 5.2 联调测试方法

**测试环境准备**：
```
1. 确认Agent API已发布
2. 前端开发服务器已启动
3. 网络连接正常
4. 测试数据准备就绪
```

**联调步骤**：
```
1. 单元测试
   - 测试单个接口调用
   - 验证数据格式
   - 检查错误处理

2. 集成测试
   - 测试完整流程
   - 前后端数据传递
   - 界面显示正确性

3. 异常测试
   - 网络异常
   - 接口超时
   - 数据异常
   - 用户输入异常
```

**问题定位技巧**：
```
1. 使用浏览器开发者工具
   - Network标签：查看请求响应
   - Console标签：查看错误日志
   - React DevTools：查看组件状态

2. 日志输出
   // 前端
   console.log('请求参数:', params);
   console.log('响应数据:', response);
   
   // Agent调试
   在灵知平台查看运行日志

3. 分段测试
   - 先测试接口直接调用
   - 再测试前端集成
   - 逐步缩小问题范围
```

### 5.3 问题定位技巧

**常见问题类型**：

**问题1：跨域错误**
```
错误信息：
Access to fetch at 'xxx' from origin 'xxx' 
has been blocked by CORS policy

解决方案：
1. 后端配置CORS
2. 使用代理服务器
3. 开发环境配置proxy
```

**问题2：数据格式不匹配**
```
现象：前端显示异常或报错

定位步骤：
1. 检查接口返回数据格式
2. 对比前端期望格式
3. 找出不匹配的字段
4. 协商统一格式

预防措施：
- 使用TypeScript定义接口类型
- 编写接口文档时提供示例
- 前后端对齐数据结构
```

**问题3：性能问题**
```
现象：响应慢、页面卡顿

定位步骤：
1. 查看Network请求耗时
2. 检查是否有重复请求
3. 分析数据量大小
4. 检查前端渲染逻辑

优化方向：
- Agent端：优化检索效率
- 接口端：数据分页、压缩
- 前端：懒加载、缓存、防抖
```

## 六、冲突解决

### 6.1 技术分歧处理

**场景1：技术方案选择**
```
示例：选择什么前端框架？

处理流程：
1. 摆出各方案优劣势
2. 结合项目需求评估
3. 必要时做技术验证
4. 集体讨论决策
5. 尊重团队决定

决策原则：
- 适用性优先
- 学习成本可控
- 团队能力匹配
```

**场景2：接口设计分歧**
```
示例：数据格式应该怎么定义？

处理流程：
1. 分析业务需求
2. 参考行业最佳实践
3. 考虑扩展性
4. 达成一致后文档化
5. 严格执行

建议：
- 简单优先
- 兼顾扩展
- 文档先行
```

### 6.2 进度滞后应对

**识别滞后**：
```
标志：
- 任务连续延期
- 每日站会总是有问题
- 团队气氛紧张

原因分析：
- 任务评估不准确
- 技术难度超预期
- 协作效率低
- 外部依赖延迟
```

**应对措施**：
```
短期应对：
1. 调整优先级，砍掉非必要功能
2. 请求讲师支援
3. 增加工作时间（适度）
4. 重新分配任务

长期改进：
1. 更准确地评估任务
2. 提前识别风险
3. 改进协作方式
4. 加强技术学习
```

### 8.2 常见问题FAQ

**Q1：任务分配不合理怎么办？**
```
A: 及时在站会提出，协商调整
   考虑成员能力和兴趣
   重新评估任务难度
```

**Q2：代码冲突频繁怎么办？**
```
A: 提高提交频率，小步快跑
   及时沟通，避免同时修改同一文件
   优化文件和模块划分
```

**Q3：进度落后怎么办？**
```
A: 分析原因，找出瓶颈
   调整优先级，聚焦核心功能
   寻求讲师支持
   适当延长时间（如需要）
```

**Q4：技术分歧无法达成一致？**
```
A: 列出各方案优劣势
   做技术验证（POC）
   必要时请讲师决策
   记录决策原因
```

**Q5：成员积极性不高怎么办？**
```
A: 了解原因（困难、兴趣、时间）
   调整任务分配
   提供帮助和支持
   营造积极氛围
```
