# 案例：35kV投资问效智能评估系统

## 一、业务背景与需求分析

### 1.1 业务背景

国网四川省电力公司面临的投资问效挑战：
- **项目数量多**：35kV及以上已投产项目数量庞大
- **数据指标复杂**：涉及投资、运行、效益等多维度指标
- **传统方式效率低**：人工分析工作量大、准确度低、周期长
- **决策支撑不足**：缺乏系统化的投后评价机制

### 1.2 核心需求

**业务目标**：
1. 自动化获取项目基本信息、投资数据、运行指标
2. 智能比对可研预测目标与实际运行数据
3. 基于规则进行8项指标评分
4. 生成结构化的投资问效分析报告

**功能需求**：
- 支持可研报告文档上传和解析
- 自动提取项目检索号并查询相关数据
- 多维度数据比对和差异分析
- 可视化展示分析结果

## 二、业务逻辑详解

### 2.1 核心评估指标体系（8项指标）

| 指标类别 | 具体指标 | 数据来源 | 评估逻辑 |
|---------|---------|---------|---------|
| **容载比** | 投运后第1/2/3年容载比实际值 | 运行指标API | 与标准值对比，判断合理性 |
| **N-1通过率** | 投运前 vs 投运当年 | 运行指标API | 改善幅度评估 |
| **设备负载率** | 投运次年关联设备负载率 | 运行指标API | 是否达到预期 |
| **负荷偏差** | 可研预测 vs 实际负荷 | 可研报告+运行指标 | 偏差率计算 |
| **最大负载率** | 投运后1/2/3年 | 运行指标API | 重过载判断 |
| **平均负载率** | 投运后1/2/3年 | 运行指标API | 利用效率评估 |
| **投资结余率** | 概算 vs 结算 | 投资数据API | 成本控制评估 |
| **建设时长** | 开工日期 → 投产日期 | 基本信息API | 工期合理性 |

### 2.2 项目分类规则

根据项目解决的问题类型，分为三类：

```
┌─────────────────────────────────────────────────────────────┐
│                     项目类型分类                             │
├──────────────┬──────────────────────────────────────────────┤
│ 经济效益类   │ 满足新增负荷需求（含解决重过载）              │
│              │ • 新增负荷供电                                │
│              │ • 解决重载/过载                               │
├──────────────┼──────────────────────────────────────────────┤
│ 安全效益类   │ 解决电网安全问题                              │
│              │ • N-1通过率改善                               │
│              │ • 网架结构优化                                │
├──────────────┼──────────────────────────────────────────────┤
│ 社会效益类   │ 满足社会民生需求                              │
│              │ • 重大活动保电                                │
│              │ • 民生改善项目                                │
└──────────────┴──────────────────────────────────────────────┘
```

### 2.3 评分规则逻辑

以**容载比评分**为例：

```
输入：
  - 投运后第1年容载比实际值: R1
  - 投运后第2年容载比实际值: R2  
  - 投运后第3年容载比实际值: R3
  - 电压等级: V（35kV/110kV等）

评分逻辑：
  1. 根据电压等级V确定容载比标准范围 [Min, Max]
  2. 判断各年容载比是否在合理区间内
  3. 综合评估得分（满分10分）：
     • 3年都在合理区间: 10分（优秀）
     • 2年在合理区间: 7分（良好）
     • 1年在合理区间: 4分（一般）
     • 0年在合理区间: 0分（不合格）

输出：
  - 容载比评分: X分
  - 评价结论: "容载比合理/偏高/偏低"
  - 建议: "建议关注第X年负荷增长情况"
```

```

      ↓
```

## 四、智能体设计架构

### 4.1 多智能体协作模式

```
┌─────────────────────────────────────────────────────────────────┐
│                     投资问效智能体系统                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌──────────────┐                                            │
│   │   用户交互层  │  ← 接收可研报告，展示分析报告              │
│   └──────┬───────┘                                            │
│          │                                                      │
│          ▼                                                      │
│   ┌──────────────┐                                            │
│   │ Task Planner │  ← 解析文档，生成执行计划(Action List)      │
│   │   (主智能体)  │                                            │
│   └──────┬───────┘                                            │
│          │                                                      │
│    ┌─────┼─────┬─────────┐                                    │
│    ▼     ▼     ▼         ▼                                    │
│ ┌────┐┌────┐┌────┐  ┌────────┐                               │
│ │数据 ││分析 ││评分 │  │ 报告   │                               │
│ │收集 ││工具 ││工具 │  │ 生成   │                               │
│ │Agent││Agent││Agent│  │ Agent  │                               │
│ └──┬─┘└──┬─┘└──┬─┘  └───┬────┘                               │

│   └──────────────┘                                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 各智能体职责定义

#### **Task Planner（任务规划器）**

**职责**：接收可研报告，生成结构化执行计划

**输入**：
- 可研报告文本内容
- 用户查询意图

**输出**：Action List（JSON数组）

```json
[
  {
    "flag": "REPORT_ANALYSIS",
    "description": "分析可研报告原文，提取关键信息",
    "depends_on": [],
    "input_hints": {
      "可研报告原文": "完整的文档解析文本"
    },
    "output_hints": {
      "检索号": "项目唯一标识",
      "主变电站名称": "",
      "投运前负荷预测值": ""
    }
  },
  {
    "flag": "GET_PROJECT_BASIC",
    "description": "获取项目基本属性信息",
    "depends_on": ["REPORT_ANALYSIS"],
    "input_hints": {
      "retrieval_no": "从解析结果中提取检索号"
    },
    "output_hints": {
      "项目名称": "",
      "电压等级": "",
      "实际投产日期": ""
    }
  },
  {
    "flag": "SCORE_JUDGE",
    "description": "基于收集数据进行8项指标评分",
    "depends_on": ["REPORT_ANALYSIS", "GET_PROJECT_BASIC", "GET_PROJECT_INVESTMENT", "GET_PROJECT_OPERATION"],
    "input_hints": {
      "可研分析结果": "REPORT_ANALYSIS的输出",
      "项目业务数据": "所有API返回的数据"
    },
    "output_hints": {
      "容载比评分": "",
      "N-1评分": "",
      "负荷偏差率评分": ""
    }
  }
]
```

#### **数据收集Agent**

**职责**：调用API获取项目相关数据

**工具**：
- `GET_PROJECT_BASIC`：获取基本信息
- `GET_PROJECT_INVESTMENT`：获取投资数据
- `GET_PROJECT_OPERATION`：获取运行指标

**执行逻辑**：
```
输入检索号 → 并行调用3个API → 整合返回数据 → 输出结构化数据
```

#### **分析工具Agent**

**职责**：文本分析和数据比对

**工具**：
- `REPORT_ANALYSIS`：可研报告关键信息提取
- `PROJECT_TYPE_EXTRACTION`：项目类型识别
- `DATA_COMPARISON`：预测vs实际数据比对

**核心能力**：
- 从非结构化文本中提取结构化字段
- 识别项目分类（经济效益/安全效益/社会效益）
- 计算偏差率和差异幅度

#### **评分工具Agent**

**职责**：执行8项指标评分

**工具**：
- `SCORE_JUDGE`：综合评分引擎

**评分维度**：
| 评分项 | 权重 | 评估标准 |
|-------|------|---------|
| 容载比 | 15% | 是否在合理区间 |
| N-1通过率 | 15% | 改善幅度 |
| 设备负载率 | 10% | 是否达到预期 |
| 负荷偏差率 | 15% | 预测准确度 |
| 最大负载率 | 10% | 重过载情况 |
| 平均负载率 | 10% | 利用效率 |
| 投资结余率 | 15% | 成本控制 |
| 建设时长 | 10% | 工期合理性 |

#### **报告生成Agent**

**职责**：整合所有分析结果，生成评估报告

**输入**：
- 项目基本信息
- 投资数据
- 运行指标
- 8项评分结果
- 差异分析结论

**输出**：结构化报告（Markdown/JSON）

## 五、核心流程设计（Plan-Execute-Judge循环）

### 5.1 闭环执行流程

```
┌────────────────────────────────────────────────────────────────┐
│                    Plan-Execute-Judge 循环                      │
├────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌──────────┐                                                 │
│   │   Start  │                                                 │
│   └────┬─────┘                                                 │
│        │                                                        │
│        ▼                                                        │
│   ┌──────────┐                                                 │
│   │   Plan   │  ← Task Planner生成Action List                 │
│   └────┬─────┘                                                 │
│        │                                                        │
│        ▼                                                        │
│   ┌──────────┐                                                 │

│        │                                                        │
│        ▼                                                        │

│   │   Judge  │  ← 判断执行结果是否满足预期                     │

│        │                                                        │
```

### 5.2 执行流程详解

**第1步：文档解析与计划生成**

```
用户上传："成都成华丛树110kV输变电工程可研报告.docx"
         ↓
Task Planner解析文档
         ↓
识别关键信息：
  - 检索号: 798-K131538-A01
  - 项目名称: 成都成华丛树110kV输变电工程
  - 预测负荷: 25MW
  - 解决问题: 满足新增负荷需求
         ↓
生成Action List（7个步骤）
```

**第2步：数据收集阶段**

```
并行执行3个API调用：

API 1: GET /api/project/basic?retrieval_no=798-K131538-A01
响应：{
  "项目名称": "成都成华丛树110kV输变电工程",
  "电压等级": 110,
  "项目类型": "经济效益类",
  "实际开工日期": "2020-11-01",
  "实际投产日期": "2022-12-01"
}

API 2: GET /api/project/investment?retrieval_no=798-K131538-A01
响应：{
  "初设概算项目总投资": 6196,
  "结算批复项目动态投资": 6309.05
}

API 3: GET /api/project/operation?retrieval_no=798-K131538-A01
响应：{
  "投运后1年实际负荷值": 18.07,
  "投运后1年最大负载率": 1.43,
  "投运后1年平均负载率": 0.03,
  "投运次年关联设备负载率": "1#丛树19.92%, 2#丛树20.09%, 1#范家坡58.71%"
}
```

**第3步：数据分析与评分**

```
数据比对：
  - 预测负荷: 25MW
  - 实际负荷: 18.07MW
  - 偏差率: (25-18.07)/25 = 27.7%
  - 评分: 偏差率>20%，得4分（满分10分）

投资分析：
  - 概算: 6196万元
  - 结算: 6309.05万元
  - 结余率: (6196-6309.05)/6196 = -1.8%
  - 评分: 结余率<0（超支），得6分
```

**第4步：报告生成**

```
整合所有信息，生成报告：

# 投资问效分析报告

## 一、项目基本信息
- 项目名称: 成都成华丛树110kV输变电工程
- 电压等级: 110kV
- 项目类型: 经济效益类
- 投产日期: 2022-12-01

## 二、投资情况
- 概算投资: 6196万元
- 结算投资: 6309.05万元
- 投资结余: -113.05万元（超支1.8%）

## 三、运行指标评估
| 指标 | 评分 | 结论 |
|-----|------|------|
| 负荷偏差率 | 4/10 | 实际负荷低于预测27.7% |
| 投资结余率 | 6/10 | 投资超支1.8% |
| ... | ... | ... |

## 四、综合评价
总体得分: 62/100
评价等级: 良好
建议: 加强负荷预测准确性，关注投资成本控制
```

## 六、关键设计要点总结

### 6.1 业务层面

1. **数据质量是关键**
   - 检索号的准确性决定后续所有数据获取
   - API数据的完整性和及时性影响评分准确度

2. **评分规则需可配置**
   - 不同电压等级的标准值不同
   - 评分权重可根据业务需求调整

3. **异常处理要完善**
   - API调用失败的重试机制
   - 数据缺失时的默认处理逻辑

### 6.2 技术层面

1. **多智能体协作模式**
   - Task Planner负责任务分解
   - 各Worker Agent专注单一职责
   - 通过依赖关系确保执行顺序

2. **Plan-Execute-Judge循环**
   - 每个Action执行后需验证结果
   - 失败时支持重试和回退
   - 记忆变量保存中间状态

3. **模块化设计**
   - 每个Agent可独立测试
   - 评分规则可热更新
   - 报告模板可自定义

## 七、与传统Agent开发的对比

| 维度 | 传统单Agent | 投资问效多Agent |
|------|------------|----------------|
| **复杂度** | 简单对话流程 | 复杂业务逻辑 |
| **数据依赖** | 单一数据源 | 多源数据整合 |
| **执行流程** | 线性执行 | 循环+条件分支 |
| **错误处理** | 简单提示 | 重试+回退机制 |
| **扩展性** | 有限 | 模块化，易扩展 |
