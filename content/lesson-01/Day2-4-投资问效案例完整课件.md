# Day2-Day4 完整课件：35kV投资问效智能评估系统

## 📋 课件概述

本课件基于国网四川省电力公司实际业务场景，通过**35kV及以上已投产项目投资问效智能评估系统**案例，系统讲解：

- **Day2**: Agent基础开发（单Agent流程搭建）
- **Day3**: 多智能体协作（Plan-Execute-Judge循环）
- **Day4**: 前端界面开发（Vibe Coding模式）

**案例特色**：
- ✅ 真实业务场景（国网四川电力试点项目）
- ✅ 完整端到端流程（文档→数据→分析→报告）
- ✅ 多智能体协作模式（Task Planner + Worker Agents）
- ✅ 复杂业务规则（8项指标评分体系）

## 第二部分：核心业务流程

### 2.1 8项评估指标体系

| 指标类别 | 具体指标 | 权重 | 数据来源 | 评分逻辑 |
|---------|---------|------|---------|---------|
| **电网运行** | 容载比 | 15% | 运行指标API | 3年都在标准区间=满分 |
| **电网运行** | N-1通过率改善 | 15% | 运行指标API | 改善幅度≥10%=满分 |
| **设备状态** | 设备负载率 | 10% | 运行指标API | 达到预期=满分 |
| **负荷预测** | 负荷偏差率 | 15% | 可研报告+运行指标 | 偏差≤10%=满分 |
| **设备状态** | 最大负载率 | 10% | 运行指标API | 无重过载=满分 |
| **设备状态** | 平均负载率 | 10% | 运行指标API | 利用效率高=满分 |
| **投资控制** | 投资结余率 | 15% | 投资数据API | 结余5-15%=满分 |
| **项目管理** | 建设时长 | 10% | 基本信息API | 按期完成=满分 |

**评分示例**（容载比）：
```
110kV项目容载比标准：[1.8, 2.2]

第1年实际值：2.0 → ✓ 在区间内
第2年实际值：2.3 → ✗ 超出上限
第3年实际值：1.9 → ✓ 在区间内

评分：2年在区间内 → 11分（满分15分）
评价：容载比基本合理，第2年偏高需关注
```

### 3.2 Plan-Execute-Judge循环详解

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    Plan-Execute-Judge 执行循环                          │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   Phase 1: PLAN（规划）                                                  │
│   ├─ Task Planner解析用户输入                                           │
│   ├─ 识别任务类型（文档分析/直接查询）                                     │
│   ├─ 生成Action List（包含依赖关系）                                      │
│   └─ 输出：结构化执行计划                                                │
│                                                                         │
│   Phase 2: EXECUTE（执行）                                               │
│   ├─ 按依赖顺序遍历Action List                                           │
│   ├─ 根据flag路由到对应Agent/工具                                        │
│   ├─ 执行具体任务（API调用/数据分析/评分计算）                            │
│   └─ 输出：执行结果 + 状态码                                             │
│                                                                         │
│   Phase 3: JUDGE（校验）                                                 │
│   ├─ 检查结果完整性（必需字段是否都存在）                                  │
│   ├─ 验证数据有效性（数值是否在合理范围）                                  │
│   ├─ 判断执行状态（成功/失败/部分成功）                                    │
│   └─ 输出：isValid + 错误信息 + 重试建议                                 │
│                                                                         │
│   ┌────────────────────────────────────────┐                           │
│   │           循环控制逻辑                  │                           │
│   │                                        │                           │
│   │   if Judge == Success:                 │                           │
│   │       → 存储结果                       │                           │
│   │       → 检查是否还有Next Action        │                           │
│   │       → Yes: 回到EXECUTE执行下一个     │                           │
│   │       → No: 进入Phase 4: GENERATE     │                           │
│   │                                        │                           │
│   │   if Judge == Fail:                    │                           │
│   │       → 记录错误日志                   │                           │
│   │       → 重试计数 + 1                   │                           │
│   │       → if 重试 < 3次:                 │                           │
│   │            → 回到EXECUTE重试           │                           │
│   │       → else:                          │                           │
│   │            → 生成错误报告              │                           │
│   │            → 进入Phase 4: GENERATE    │                           │
│   │                                        │                           │
│   └────────────────────────────────────────┘                           │
│                                                                         │
│   Phase 4: GENERATE（生成）                                              │
│   ├─ 整合所有成功执行的结果                                              │
│   ├─ 调用Report Generator生成报告                                        │
│   ├─ 格式化输出（Markdown + 结构化数据）                                 │
│   └─ END                                                                 │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.3 关键设计模式

#### 模式1: 依赖驱动执行

```javascript
// Action依赖关系定义
const actions = [
  { flag: 'A', depends_on: [] },           // 无依赖，立即执行
  { flag: 'B', depends_on: ['A'] },        // 依赖A，A完成后执行
  { flag: 'C', depends_on: ['A'] },        // 依赖A，A完成后执行
  { flag: 'D', depends_on: ['B', 'C'] },   // 依赖B和C，两者都完成后执行
];

// 执行顺序：A → (B, C并行) → D
```

#### 模式2: 结果校验与重试

```javascript
// Judge逻辑伪代码
function judge(action, result) {
  // 1. 检查必需字段
  const requiredFields = action.expected_output;
  const missing = requiredFields.filter(f => !result[f]);
  
  if (missing.length > 0) {
    return {
      valid: false,
      error: `缺少必需字段: ${missing.join(', ')}`,
      canRetry: result.retryCount < 3
    };
  }
  
  // 2. 数据有效性检查
  if (result.score < 0 || result.score > 100) {
    return {
      valid: false,
      error: '评分超出有效范围',
      canRetry: true
    };
  }
  
  return { valid: true };
}
```

#### 模式3: 记忆变量传递

```javascript
// 在灵搭中通过代码块实现状态传递
function userFunction(param) {
  const state = JSON.parse(param['currentState'] || '{}');
  const newResult = JSON.parse(param['newResult']);
  
  // 更新状态
  state.results = state.results || {};
  state.results[newResult.step] = newResult.data;
  state.completedSteps = (state.completedSteps || 0) + 1;
  
  return {
    updatedState: JSON.stringify(state),
    progress: `${state.completedSteps}/${state.totalSteps}`
  };
}
```

#### ✅ API集成要点

```javascript
// 推荐的API调用封装
const api = {
  // 1. 配置集中管理
  config: {
    baseUrl: import.meta.env.VITE_API_BASE_URL,
    uuid: import.meta.env.VITE_AGENT_UUID,
    authKey: import.meta.env.VITE_AUTH_KEY,
  },
  
  // 2. 统一请求处理
  async callAgent(query, context = {}) {
    const response = await fetch(`${this.config.baseUrl}/${this.config.uuid}`, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${this.config.authKey}`,
      },
      body: JSON.stringify({ query, context }),
    });
    
    if (!response.ok) {
      throw new Error(`HTTP ${response.status}: ${response.statusText}`);
    }
    
    const result = await response.json();
    return this.parseResponse(result.data.answer);
  },
  
  // 3. 响应解析
  parseResponse(answer) {
    // 尝试从Markdown中提取JSON
    const jsonMatch = answer.match(/```json\n([\s\S]*?)\n```/);
    if (jsonMatch) {
      return JSON.parse(jsonMatch[1]);
    }
    // 返回原始文本
    return { report: answer };
  }
};
```

## 第六部分：常见问题与解决方案

### 6.1 Day2常见问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 文档解析失败 | 文件格式不支持 | 确保.docx/.pdf格式，不要加密 |
| 信息提取不完整 | 提示词不够精确 | 增加字段示例，优化提示词 |
| HTTP调用400错误 | 参数名错误 | 检查参数名与API文档一致 |
| 数据整合出错 | JSON解析失败 | 添加try-catch，验证数据格式 |
| 流程不执行 | 连接条件未满足 | 检查激活条件，确保前置模块成功 |

### 6.2 Day3常见问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| Task Planner输出非JSON | 提示词约束不足 | 强调"仅输出JSON"，提供示例 |
| 子Agent无法调用 | 缺少Agent对话结束模块 | 确保子Agent包含"Agent对话结束" |
| 依赖执行顺序错乱 | Action依赖定义错误 | 检查depends_on是否正确 |
| 循环模块不执行 | 输入非数组类型 | 确保输入为JSON数组 |
| 结果校验总是失败 | 校验规则太严格 | 放宽校验条件，允许部分字段缺失 |

### 6.3 Day4常见问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| API调用跨域错误 | CORS限制 | 配置代理或使用后端转发 |
| Markdown渲染异常 | 特殊字符未转义 | 使用react-markdown组件 |
| 雷达图数据不显示 | 数据格式错误 | 确保数据为[{item, score}]格式 |
| 界面样式错乱 | CSS冲突 | 使用CSS Modules或styled-components |
| 构建失败 | 依赖版本冲突 | 检查package.json，锁定版本 |
