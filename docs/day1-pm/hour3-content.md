# 课时3: 工具演进与封装

**时长**: 40分钟  
**主题**: 从API到Skills的工具演进路径与完整技术栈架构

---

## 1. API → MCP: 描述标准化 (10分钟)

### 讲解要点

**问题的提出**:
> "Agent有了Acting的能力，但如何让AI理解和调用这些工具？"

**API时代的局限**:

| 问题 | 说明 | 影响 |
|------|------|------|
| **格式混乱** | 每个API格式各异 | AI无法统一理解 |
| **文档复杂** | Swagger文档给人看 | AI读不懂 |
| **调用方式** | 需要人写代码 | AI无法自主调用 |
| **上下文传递** | 参数和返回格式不统一 | 难以串联多个工具 |

**类比**:
> "API就像各种电器的插头——美标、欧标、国标，各不相同。你要用就得买对应的转换器。"

---

**MCP的突破 - 描述标准化**:

**MCP (Model Context Protocol) 核心思想**:
> "让工具具备'自描述'能力，AI能直接读懂工具能做什么。"

**MCP的关键特性**:

1. **自描述 (Self-describing)**
   - 工具自带结构化说明书
   - AI能读懂：名称、功能、参数、返回值

2. **标准化 (Standardized)**
   - 统一格式
   - 统一调用方式
   - 统一错误处理

3. **生态互通 (Interoperable)**
   - 任何遵循MCP的工具，AI都能用
   - 跨平台、跨系统

**MCP描述示例**:

```json
{
  "name": "query_investment_data",
  "description": "查询指定时间范围内的投资项目数据",
  "parameters": {
    "type": "object",
    "properties": {
      "region": {
        "type": "string",
        "description": "查询区域，如'成都高新区'"
      },
      "start_date": {
        "type": "string", 
        "description": "开始日期，格式YYYY-MM-DD"
      },
      "end_date": {
        "type": "string",
        "description": "结束日期，格式YYYY-MM-DD"
      }
    },
    "required": ["region", "start_date", "end_date"]
  },
  "returns": {
    "type": "array",
    "items": {
      "type": "object",
      "properties": {
        "project_id": {"type": "string"},
        "investment": {"type": "number"},
        "roi": {"type": "number"}
      }
    }
  }
}
```

**AI如何理解**:
```
AI读取MCP描述:
- 名称: query_investment_data
- 功能: 查询投资项目数据
- 需要参数: region, start_date, end_date
- 返回: 项目列表，包含investment和roi

AI决策:
"用户要查投资数据，这个工具正好匹配！"
```

---

**两层价值**:

| 层次 | 核心 | 解决的问题 |
|------|------|-----------|
| **第一层** | 描述API让AI看懂 | AI能理解单个工具 |
| **第二层** | 形成标准实现生态 | 任何MCP工具AI都能用 |

**核心洞察**:
> "MCP不是'能不能'的问题，是'通不通'的问题。它让工具像USB-C一样通用。"

### 讲师话术示例

**讲问题**:
> "前面讲了Acting的四种方式，但有个关键问题：这些工具怎么让AI理解和调用？在API时代，每个接口格式都不一样，文档是给人看的，AI看不懂。就像你面前有10个电器，10种不同的插头，你不知道该用哪个。"

**讲MCP突破**:
> "MCP就是解决这个问题。它让工具具备'自描述'能力——工具自己带一张'说明书'，而且是AI能读懂的说明书。AI一看就知道：这个工具能做什么，需要什么参数，返回什么结果。"

**讲两层价值**:
> "MCP有两层价值。第一层，描述API让AI看懂，这是基础。第二层，也是更重要的，形成标准，实现生态互通。就像USB-C，任何厂商的USB-C设备都能插到你的手机上。MCP让任何遵循标准的工具，AI都能直接用。"

### 白板/板书建议

**演进对比**:
```
API时代:
工具A ← 文档A(人读)
工具B ← 文档B(人读)  → 人翻译 → AI
工具C ← 文档C(人读)

MCP时代:
工具A ← MCP描述(AI读)
工具B ← MCP描述(AI读)  → AI直接理解
工具C ← MCP描述(AI读)
```

**关键标注**:
> MCP = 工具的"USB-C标准"

## 2. Tools → Skills: 语义封装 (15分钟)

### 讲解要点

**MCP解决了调用问题，但还不够**:

**Tools的局限**:

| 局限 | 说明 | 示例 |
|------|------|------|
| **粒度太细** | 一个Tool只做一件事 | query_data只能查数据 |
| **语义太技术** | 技术命名，业务含义不明 | calculate_roi vs 计算投资回报率 |
| **缺乏业务上下文** | 不知道业务规则 | 不知道权限、流程 |
| **难以组合** | 多个Tools需要手动串联 | 查数据→分析→生成报告 |

---

**Skills: 从"技术"到"业务"的升维**:

**Skills的定义**:
> "面向AI的业务能力封装，让AI能理解和组合使用。"

**三层封装架构**:

```
Skills (业务语义层)
    ├── 命名: 业务语义 ("查询投资效益")
    ├── 功能: 完整业务场景
    ├── 组合: 可与其他Skills组合
    └── 描述: AI能理解的业务说明

Code Tools (代码封装层)  
    ├── 自定义代码逻辑
    ├── 封装多个Tools
    └── 实现特定功能

Tools (原子能力层)
    ├── 原子操作
    ├── 技术实现
    └── 具体函数
```

**对比示例**:

**Tools层** (原子能力):
```python
# Tool 1: 查询数据
def query_data(table, conditions):
    return db.query(table, conditions)

# Tool 2: 计算ROI  
def calculate_roi(investment, returns):
    return (returns - investment) / investment

# Tool 3: 生成图表
def generate_chart(data, chart_type):
    return chart_lib.create(data, chart_type)
```

**Code Tools层** (代码封装):
```python
# 封装ROI计算流程
def analyze_roi_tool(project_id):
    data = query_data("investments", {"id": project_id})
    roi = calculate_roi(data["investment"], data["returns"])
    chart = generate_chart([roi], "bar")
    return {"roi": roi, "chart": chart}
```

**Skills层** (业务语义):
```json
{
  "name": "analyze_investment_effectiveness",
  "description": "分析投资项目的效益情况，计算ROI并生成可视化报告。适用于投资问效场景。",
  "parameters": {
    "project_id": "项目ID",
    "include_benchmark": "是否包含行业基准对比"
  },
  "returns": {
    "roi": "投资回报率",
    "chart": "可视化图表",
    "suggestion": "投资建议"
  }
}
```

**关键差异**:

| 维度 | Tools | Skills |
|------|-------|--------|
| **粒度** | 原子操作 | 业务场景 |
| **命名** | 技术术语 | 业务语义 |
| **复杂度** | 单一功能 | 可组合、可编排 |
| **面向** | 程序员 | AI Agent |
| **示例** | query_data() | "查询投资效益" |

---

**Skills的设计原则**:

1. **单一职责** (但面向业务)
   - 一个Skill只做一件业务事
   - 示例: "查询投资效益"而不是"查询数据并计算并生成图表"

2. **语义自描述**
   - 名称和描述让AI能懂业务含义
   - 示例: analyze_investment_effectiveness vs query_data

3. **可组合性**
   - 多个Skills可以串联使用
   - 示例: 查询→分析→生成报告

4. **可复用性**
   - 不同场景可复用同一个Skill
   - 示例: "查询投资效益"可用于月报、年报、专项分析

**Skills组合示例**:
```
复杂任务: "分析成都高新区投资项目并生成报告"

Skills组合:
1. query_investment_data (查询投资数据)
2. analyze_roi (分析ROI)
3. generate_report (生成报告)

Agent编排:
R: 需要分析投资效益
P: 分三步：查询→分析→生成
A: 依次调用三个Skills
Ref: 报告生成完成
```

### 核心观点

> "Tools解决'怎么做'，Skills解决'做什么'——让AI能**理解业务语义**并**组合使用**。"

### 讲师话术示例

**讲Tools局限**:
> "MCP让AI能看懂工具了，但还有一个问题。Tools的粒度太细，一个Tool只做一件事，比如query_data只能查数据。而且命名都是技术术语，比如calculate_roi，AI知道这是计算ROI，但不知道在什么业务场景下用。更麻烦的是，一个完整任务需要多个Tools串联，AI不知道怎么组合。"

**讲Skills升维**:
> "Skills就是解决这个问题。它是面向业务的封装。比如不再叫query_data，而是叫'查询投资效益'。这个Skill内部可能用了三个Tools：查数据、算ROI、生成图表。但对外暴露的是业务语义。AI一看就知道：哦，这是做投资分析的，我可以用它。"

**讲组合性**:
> "更重要的是，Skills可以组合。比如分析投资效益这个任务，可以拆成三个Skills：查询数据、分析ROI、生成报告。Agent通过RPAR循环来编排：先调用查询，再调用分析，最后调用生成。这就像搭积木，每个Skill是一块积木，可以拼出各种业务场景。"

### 白板/板书建议

**三层封装阶梯图**:
```
      Skills
   (业务语义封装)
        ↑ 语义化
   Code Tools
   (代码逻辑封装)
        ↑ 代码化
      Tools
   (原子能力)
        ↑ MCP协议
       API
   (原始接口)
```

**或者金字塔图**:
```
         /
        / Skills (业务)
       /─────────
      / Code Tools (代码)
     /─────────
    / Tools (原子)
   /─────────
  / API (原始)
 /─────────
```

## 3. 完整技术栈架构 (15分钟)

### 讲解要点

**从底层到上层的完整栈**:

```
┌─────────────────────────────────────────┐
│  第5层: RPAR循环 (智能编排)              │
│  Reasoning → Planning → Acting → Reflection│
│  Agent的"大脑" - 决策和编排              │
└──────────────────┬──────────────────────┘
                   ↓ 调用
┌─────────────────────────────────────────┐
│  第4层: Skills (业务语义封装)            │
│  "查询投资效益" "生成报告" "分析ROI"      │
│  AI能理解的业务能力单元                  │
└──────────────────┬──────────────────────┘
                   ↓ 内部使用
┌─────────────────────────────────────────┐
│  第3层: Code Tools (代码封装)            │
│  自定义代码逻辑                          │
│  封装多个Tools                           │
└──────────────────┬──────────────────────┘
                   ↓ 调用
┌─────────────────────────────────────────┐
│  第2层: Tools (原子能力)                 │
│  query_data, calculate_roi, generate_chart│
│  具体的功能函数                          │
└──────────────────┬──────────────────────┘
                   ↓ 通过
┌─────────────────────────────────────────┐
│  第1层: MCP (标准化协议)                 │
│  统一描述格式和调用方式                  │
│  让AI能理解工具                          │
└──────────────────┬──────────────────────┘
                   ↓ 调用
┌─────────────────────────────────────────┐
│  第0层: API (原始能力)                   │
│  各系统的原始接口                        │
│  数据的源头                              │
└─────────────────────────────────────────┘
```

**各层职责**:

| 层级 | 名称 | 职责 | 解决的问题 |
|------|------|------|-----------|
| 5 | RPAR | 决策编排 | 什么时候做什么 |
| 4 | Skills | 业务语义 | 做什么业务 |
| 3 | Code Tools | 代码逻辑 | 怎么组织代码 |
| 2 | Tools | 原子能力 | 怎么实现功能 |
| 1 | MCP | 协议标准 | 怎么让AI理解 |
| 0 | API | 原始能力 | 有什么能力 |

---

**与灵知平台的映射**:

| 技术概念 | 灵知平台对应 |

| RPAR | 编排流程图 |

| Code Tools | 代码块模块 |

| MCP | 模块配置描述 |

```
[用户提问] → [信息分类] → [条件判断]
                    ↓
        ┌──────────┼──────────┐
        ↓          ↓          ↓
    [查询Skill] [分析Skill] [报告Skill]
        ↓          ↓          ↓
    [HTTP调用]  [代码块]   [智能对话]
        ↓          ↓          ↓
    [外部API]  [本地计算]  [生成回复]
```

---

**演进逻辑总结**:

```
【问题】AI需要工具，但如何让AI理解和使用？
    ↓
【暴露】API: 能力可以被调用，但AI看不懂
    ↓
【标准化】MCP: 自描述能力，AI能理解了
    ↓
【封装】Skills: 业务语义化，AI能组合使用了
    ↓
【编排】RPAR: 自主决策，循环优化
    ↓
【结果】Agentic AI: 能自主执行任务的智能体
```

### 核心金句

> "从API到RPAR，本质上是在回答一个问题：如何让AI从'能访问工具'到'能理解和使用工具'再到'能自主编排工具'。"

### 讲师话术示例

**总结演进**:
