## 1. RPAR的裁剪与实例化 (10分钟)

### 讲解要点

**核心概念**:
> "RPAR是'完全体'，但不同场景可以用'裁剪版'。"

**三种主要裁剪方式**:

#### 1.1 ReAct: R-A-R (简化版)

**结构**: Reasoning → Acting → Reasoning (循环)

**特点**:
- 去掉了显式Planning
- 去掉了显式Reflection
- 把规划和反思隐含在Reasoning中

**适用场景**:
- 简单任务
- 不需要复杂规划
- 边想边做即可

**示例**:
```
用户: "查一下今天的天气"

ReAct循环:
Thought: 用户需要天气信息，我需要调用天气API
Action: 调用get_weather API
Observation: 获得天气数据
Thought: 数据已获取，可以回答用户了
Answer: "今天多云，15-22°C..."
```

#### 1.2 Plan-and-Execute: R-P-A (规划版)

**结构**: Reasoning → Planning → Acting(List)

**特点**:
- 显式Planning阶段
- Acting批量执行（List表示一次性执行多个）
- 去掉了显式Reflection

**适用场景**:
- 步骤明确的任务
- 需要提前规划
- 批量执行更高效

**示例**:
```
用户: "帮我订一张北京到上海的机票，要上午的"

Planning阶段:
1. 查询航班信息
2. 筛选上午航班
3. 预订选中的航班

Acting(List):
- 调用query_flights
- 调用filter_morning
- 调用book_ticket
```

#### 1.3 完整RPAR: R-P-A-R (完全体)

**结构**: Reasoning → Planning → Acting → Reflection (循环)

**特点**:
- 四个环节完整
- 支持试错和修正
- 最灵活但最复杂

**适用场景**:
- 复杂任务
- 需要试错调整
- 不确定性高

**示例**:
```
用户: "分析一下成都高新区的投资项目效益"

Round 1:
R: 需要查询投资数据
P: 分三步：查询→分析→报告
A: 调用query_investment
Ref: 数据获取成功，但发现不完整

Round 2:
R: 需要补充查询其他数据源
P: 查询补充数据
A: 调用query_supplementary
Ref: 数据完整了

Round 3:
R: 可以生成报告了
...
```

### 对比总结表

| 范式 | 结构 | 特点 | 适用场景 |
|------|------|------|----------|
| **ReAct** | R-A-R | 简单直接 | 简单查询类任务 |
| **Plan-and-Execute** | R-P-A | 先规划后执行 | 步骤明确的任务 |
| **完整RPAR** | R-P-A-R | 完整循环 | 复杂、需试错的任务 |

### 讲师话术示例

**引入裁剪概念**:
> "刚才讲了RPAR的四个环节，但你们可能会想：每个任务都要走完整四个环节吗？那太麻烦了。实际上，不同场景可以用不同的'裁剪版'。就像你去超市买菜，不需要写详细的购物清单（Planning），直接看想买什么就拿什么。但如果你装修房子，就需要先规划好要买什么。"

**讲解ReAct**:
> "最常用的裁剪版叫ReAct，它只保留了Reasoning和Acting，把Planning和Reflection藏在Reasoning里。简单说就是'边想边做'。比如用户问天气，Agent想一下需要什么数据，然后直接去查，查到就回答。不需要复杂的规划。"

**讲解Plan-and-Execute**:
> "但如果任务比较复杂，比如订机票，就需要Plan-and-Execute。先规划好：查航班→选航班→预订。然后再一步步执行。这样效率更高，不会走一步看一步。"

**讲解完整RPAR**:
> "最复杂的情况，比如投资分析，就需要完整RPAR。因为你可能第一次查的数据不完整，需要反思，然后重新规划，再执行。这就是试错和修正的过程。"

### 白板/板书建议

**三种裁剪对比**:
```
ReAct:          R → A → R
               (简捷)

Plan-and-Exec:  R → P → A
               (先规划)

完整RPAR:       R → P → A → Ref
               (全功能)
```

## 2. 范式对比与选择 (10分钟)

### 讲解要点

**选择决策树**:

```
任务是否简单明确？
├── 是 → ReAct
└── 否 → 步骤是否明确？
          ├── 是 → Plan-and-Execute
          └── 否 → 完整RPAR
```

**详细对比**:

| 维度 | ReAct | Plan-and-Execute | 完整RPAR |
|------|-------|------------------|----------|
| **复杂度** | 低 | 中 | 高 |
| **效率** | 中 | 高 | 低 |
| **灵活性** | 中 | 低 | 高 |
| **容错性** | 低 | 低 | 高 |
| **适用** | 简单问答 | 标准流程 | 探索性任务 |

**电力行业场景匹配**:

| 场景 | 推荐范式 | 理由 |
|------|---------|------|
| 查询用电数据 | ReAct | 单次查询，简单直接 |
| 月度报表生成 | Plan-and-Execute | 步骤固定：取数→计算→生成 |
| 故障诊断分析 | 完整RPAR | 需要试错，不确定因素多 |

### 核心观点

> "没有最好的范式，只有最适合的范式。简单任务用ReAct，标准流程用Plan-and-Execute，复杂探索用完整RPAR。"

### 讲师话术示例

**讲解决策**:
> "那怎么选择呢？给大家一个简单的决策树。首先看任务是否简单明确。如果只是查个数据、问个问题，用ReAct就够了。如果任务比较复杂，再看步骤是否明确。如果步骤很清楚，比如生成报表就是取数-计算-生成三步，那就用Plan-and-Execute。如果步骤不明确，需要边做边看，比如故障诊断，可能查了这个数据发现不对，还要查别的，那就用完整RPAR。"

**电力场景举例**:
> "在电力行业，查询用电数据用ReAct，直接问直接答。月度报表生成用Plan-and-Execute，步骤固定。但故障诊断就要用完整RPAR，因为你可能查了温度数据发现没问题，还要查振动数据，一步步排查。"

### 白板/板书建议

**决策树**:
```
任务复杂度
    ↓
简单? → ReAct
    ↓
步骤明确?
    ↓
是 → Plan-and-Execute
    ↓
否 → 完整RPAR
```

## 3. Acting详解 - Agent的"手脚" (20分钟)

### 讲解要点

**核心概念**:
> "RPAR是'大脑'，Acting是'手脚'。没有Acting，Agent只是聊天机器人。"

**Acting四象限**:

| 类别 | 作用 | 典型工具 | 示例 |
|------|------|---------|------|
| **API调用** | 调用外部服务 | REST API, GraphQL | 查询数据库、发送通知 |
| **代码执行** | 运行本地代码 | Python, JS | 数据分析、复杂计算 |
| **文件操作** | 读写文件 | 文件系统 | 生成报告、保存日志 |
| **其他工具** | 特殊能力 | 浏览器、数据库 | 网页抓取、SQL查询 |

#### 3.1 API调用

**定义**: 通过HTTP/HTTPS调用外部服务接口

**适用场景**:
- 查询远程数据
- 调用第三方服务
- 触发远程操作

**示例**:
```python
# 查询用电数据
response = requests.post(
    "https://api.power.com/query",
    json={"region": "成都高新区", "date": "2024-03-01"}
)
data = response.json()
```

**特点**:
- 最常用的Acting方式
- 依赖网络
- 需要API文档

#### 3.2 本地代码执行

**定义**: 在本地环境执行代码片段

**适用场景**:
- 复杂数据处理
- 数学计算
- 算法执行
- 格式转换

**示例**:
```python
# 计算ROI
def calculate_roi(investment, returns):
    return (returns - investment) / investment * 100

roi = calculate_roi(1000000, 1200000)
```

**特点**:
- 灵活性高
- 可处理复杂逻辑
- 安全性需要注意

#### 3.3 文件系统操作

**定义**: 读写本地文件

**适用场景**:
- 生成报告文件
- 读取配置文件
- 保存日志
- 批量处理文件

**示例**:
```python
# 生成Excel报告
import pandas as pd

df = pd.DataFrame(data)
df.to_excel("report.xlsx", index=False)
```

**特点**:
- 持久化存储
- 便于人工查看
- 适合批量处理

#### 3.4 其他工具

**浏览器操作**:
- 网页抓取
- 自动化测试
- 填写表单

**数据库直连**:
- SQL查询
- 数据更新
- 事务操作

**命令行执行**:
- 系统命令
- 脚本执行
- 文件操作

### 核心观点

> "Acting让Agent从'大脑'变成'全人'——既能思考，又能动手。四种Acting方式涵盖了Agent与外部世界交互的主要途径。"

### 讲师话术示例

**引入Acting**:
> "讲了半天RPAR，你们可能会问：Agent的思考和规划，最后怎么落地？答案就是Acting——执行。Acting是Agent的'手脚'，让Agent从'动口'变成'动手'。"

**讲解四类**:
> "Acting主要有四类。第一类，API调用，也是最常用的。Agent要查数据，就调用数据查询API；要发通知，就调用消息推送API。第二类，代码执行，在本地运行Python或JavaScript代码，做复杂的数据处理或计算。第三类，文件操作，读写文件，比如生成Excel报告。第四类，其他工具，比如操作浏览器、直接连数据库等。"

**强调重要性**:
> "没有Acting，Agent只是聊天机器人，只能给你建议。有了Acting，Agent才能真正帮你完成任务。这是Agent与传统AI应用的本质差别。"

### 白板/板书建议

**Acting四象限图**:
```
              在线/远程
                  │
    API调用       │       其他工具
  (调用服务)      │     (浏览器等)
                  │
──────────────────┼──────────────────
                  │
   代码执行       │      文件操作
  (本地计算)      │    (读写文件)
                  │
              本地/离线
```

或者简化为2x2网格:
```
┌─────────────┬─────────────┐
│  API调用    │  其他工具   │
│  (外部服务) │ (浏览器等)  │
├─────────────┼─────────────┤
│  代码执行   │  文件操作   │
│  (本地计算) │ (文件系统)  │
└─────────────┴─────────────┘
```
