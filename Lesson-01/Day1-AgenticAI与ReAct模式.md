**1. AI发展的三个阶段（15分钟）**

**PPT大纲**：

**Slide 1：AI发展的三个阶段**
```
阶段一：Rule-Based AI（基于规则）
├── 特点：硬编码规则，if-else逻辑
├── 局限：无法处理复杂、多变的场景
└── 例子：早期专家系统

阶段二：Machine Learning AI（机器学习）
├── 特点：从数据中学习模式
├── 局限：需要大量标注数据，泛化能力有限
└── 例子：图像识别、语音识别

阶段三：Agentic AI（智能体AI）⭐
├── 特点：自主决策、工具使用、持续学习
├── 优势：灵活应对复杂任务，可扩展性强
└── 例子：AutoGPT、灵知平台Agent
```

**讲师讲解要点**：

**阶段一：Rule-Based AI（1990s-2000s）**
- 电力行业早期也用过这类系统
- 例子：设备故障诊断系统
  - 规则：如果温度>80°C且振动>5mm/s → 报警
  - 问题：规则太多难以维护，新故障类型无法识别

**阶段二：Machine Learning AI（2010s-2020s）**
- 电力行业的应用：负荷预测、故障预测
- 例子：基于历史数据预测变压器故障
- 局限：
  - 需要大量历史故障数据（但故障样本少）
  - 模型是"黑盒"，难以解释决策过程
  - 只能完成单一任务，无法串联复杂流程

**阶段三：Agentic AI（2023-至今）⭐**
- 核心理念：AI不仅能"思考"，还能"行动"
- 关键能力：
  1. **自主性**：不需要人类一步步指令
  2. **规划能力**：能分解复杂任务
  3. **工具使用**：能调用外部API、数据库等
  4. **记忆能力**：能记住上下文和历史

**思考互动**：
> 在电力行业，你们觉得哪些工作最适合用Agent来完成？为什么？

**引导方向**：
- 需要多步骤决策的任务（如故障排查）
- 需要调用多个系统的任务（如查询数据+生成报告）
- 需要灵活应对的任务（如客户咨询）

**2. Agent的四大核心特征（25分钟）**

**PPT大纲**：

**Slide 2：Agent的四大核心特征**
```
┌─────────────────────────────────────────────────────────────┐
│                    Agentic AI 核心特征                       │
├──────────────┬──────────────┬──────────────┬────────────────┤
│   自主性     │   规划能力    │   工具使用    │    记忆能力    │
│ Autonomy     │  Planning    │   Tool Use   │    Memory      │
├──────────────┼──────────────┼──────────────┼────────────────┤
│ 独立决策     │ 任务分解     │ 调用外部能力  │ 上下文管理     │
│ 无需人工     │ 步骤规划     │ API/数据库    │ 历史记录       │
│ 逐步指令     │ 动态调整     │ 搜索引擎等    │ 知识积累       │
└──────────────┴──────────────┴──────────────┴────────────────┘
```

**详细讲解**：

**特征1：自主性（Autonomy）**
- 传统AI：输入→输出（一次性）
- Agentic AI：输入→思考→行动→观察→再思考...（循环）
- **案例对比**：
  - 传统：用户问"今天成都会下雨吗？"→ AI直接回答
  - Agent：用户问"今天成都会下雨吗？"→ Agent判断需要查询天气API → 调用API → 获取数据 → 回答用户

**特征2：规划能力（Planning）**
- 能自动将复杂任务分解为子任务
- 能根据执行反馈动态调整计划
- **案例演示**：
  ```
  用户请求："帮我查一下成都地区近7天的用电趋势，并生成一个可视化图表"
  
  Agent的规划过程：
  Step 1: 理解需求 → 需要查询用电数据 + 生成图表
  Step 2: 查询数据 → 调用用电数据查询接口
  Step 3: 数据分析 → 计算平均值、峰值、趋势
  Step 4: 生成图表 → 调用图表生成工具
  Step 5: 展示结果 → 返回图表和文字说明
  ```

**特征3：工具使用（Tool Use）**
- Agent不是孤立的，可以调用各种外部工具
- 工具类型：
  - **数据查询工具**：数据库查询、API调用
  - **计算工具**：数学计算、数据分析
  - **生成工具**：图表生成、文档生成
  - **外部服务**：天气查询、地图服务等
- **电力行业工具示例**：
  - 设备状态查询API
  - 历史负荷数据接口
  - 工单系统接口
  - 知识库检索接口

**特征4：记忆能力（Memory）**
- **短期记忆**：当前对话的上下文
- **长期记忆**：历史对话、用户偏好、领域知识
- **案例**：
  ```
  用户第1轮："查询变压器T-001的状态"
  Agent：返回状态信息
  
  用户第2轮："它的温度是多少？"
  传统AI：不理解"它"指什么
  Agent：通过短期记忆知道"它"=变压器T-001，直接返回温度
  ```

#### 课程结构

**1. ReAct 模式简介（15分钟）**

**PPT大纲**：

**Slide 4：什么是 ReAct？**
```
ReAct = Reasoning（推理）+ Acting（行动）

核心思想：
让AI像人类一样思考：
先思考（Thought）→ 再行动（Action）→ 观察结果（Observation）→ 再思考...

形成循环，直到任务完成
```

**讲师讲解**：

**ReAct的起源**：
- 来自论文《ReAct: Synergizing Reasoning and Acting in Language Models》（2022）
- 作者：Google Research团队
- 核心发现：让模型显式地"思考"再"行动"，比直接生成答案效果更好

**为什么需要ReAct？**

**问题1：传统LLM的局限性**
```
用户问："成都今天的气温比昨天高多少？"

传统LLM的回答：
"根据我的训练数据，我无法获取实时天气信息..."

问题：LLM只有训练数据，无法获取实时信息
```

**问题2：简单的工具调用也不够**
```
用户问："帮我查一下成都地区用电负荷最大的前3个区域，
          并分析可能的原因"

如果只是调用工具：
- 工具返回原始数据
- 但用户需要的是"分析原因"
- 这需要多步骤推理
```

**ReAct的解决方案**：
```
循环机制：Thought → Action → Observation → Thought → ...

Step 1 - Thought（思考）：
"用户需要查询成都地区的用电负荷数据，
 然后找出最大的3个区域，最后分析原因。
 我需要先调用数据查询工具。"

Step 2 - Action（行动）：
调用工具：query_power_load(city="成都", date="今天")

Step 3 - Observation（观察）：
获取到数据：[{"area": "高新区", "load": 850MW}, ...]

Step 4 - Thought（再思考）：
"数据中高新区负荷最高，其次是天府新区。
 高新区负荷高的原因可能是：
 1. 产业园区集中
 2. 高温天气空调负荷大
 让我整理这个分析结果。"

Step 5 - Action（最终行动）：
生成回答，包含数据和分析
```

#### 2. ReAct 循环详解（25分钟）

**PPT大纲**：

**Slide 5：ReAct 循环流程图**
```
         ┌─────────────────────────────────────────────────────┐
         │                                                     │
         ▼                                                     │
   ┌──────────┐    ┌──────────┐    ┌──────────┐              │
   │  Thought │───▶│  Action  │───▶│Observation│──────────────┘
   │  （思考） │    │  （行动） │    │ （观察）  │
   └──────────┘    └──────────┘    └──────────┘
         │
         ▼（当任务完成时）
   ┌──────────┐
   │  Answer  │
   │  （回答） │
   └──────────┘
```

**详细讲解每个环节**：

**Thought（思考）**：
- 对当前状态的分析
- 对下一步行动的计划
- 是否已经完成任务的判断
- **关键**：必须是显式的、可读的推理过程

**Action（行动）**：
- 具体的工具调用
- 参数的准备
- 格式通常是JSON或特定格式
- **例子**：
  ```json
  {
    "action": "query_database",
    "parameters": {
      "table": "power_load",
      "city": "成都",
      "date": "2024-03-01"
    }
  }
  ```

**Observation（观察）**：
- 工具返回的结果
- 环境反馈的信息
- 新的数据或状态
- **注意**：观察结果会作为下一步思考的输入

**循环终止条件**：
1. 任务已完成（得到最终答案）
2. 达到最大循环次数（防止无限循环）
3. 发生错误无法继续

**1. MCP 协议简介（20分钟）**

**PPT大纲**：

**Slide 9：什么是 MCP？**
```
MCP = Model Context Protocol（模型上下文协议）

核心作用：标准化Agent与外部世界的交互方式

类比：
- HTTP是浏览器与服务器的通信协议
- MCP是AI模型与工具/数据的通信协议
```

**讲师讲解**：

**为什么需要MCP？**

**问题背景**：
- 不同的AI模型、不同的工具，接口格式各异
- 开发者需要为每个工具写适配代码
- 工具之间难以组合和复用

**MCP的解决方案**：
- 统一的接口规范
- 标准化的上下文传递
- 支持工具的组合和编排

**MCP的核心组件**：

```
┌────────────────────────────────────────────────────────────┐
│                     MCP 架构                                │
├──────────────┬──────────────────────────────┬──────────────┤
│   上下文层   │        协议层                │    工具层    │
├──────────────┼──────────────────────────────┼──────────────┤
│ 对话历史     │ 标准化的请求/响应格式        │ 工具描述     │
│ 用户偏好     │ 统一的参数传递规范           │ 工具调用     │
│ 环境信息     │ 错误处理机制                 │ 结果返回     │
└──────────────┴──────────────────────────────┴──────────────┘
```

**MCP在ReAct中的作用**：
- **标准化Action**：所有工具调用都遵循统一格式
- **上下文传递**：Observation自动成为下一步Thought的输入
- **错误处理**：统一的错误码和恢复机制

```
选题1：智能问数

选题2：智能写作
├── 场景：公文、报告、邮件的智能生成

选题3：工单处理

├── 前端功能：工单列表、详情展示、处理建议面板
└── 复杂度：⭐⭐⭐⭐
```

**详细讲解各选题**：

**选题1：智能问数**

**业务场景**：
- 领导问："这个月各区域的用电量排名"
- 分析师问："对比一下去年同期，哪些区域增长最快？"
- 运维问："昨天半夜为什么负荷突然升高？"

**ReAct设计要点**：
```
Thought 1: 理解用户意图
  - 实体识别：时间（这个月）、指标（用电量）、维度（各区域）
  - 操作识别：排名、对比

Action 1: 调用数据查询Skill
  - 参数：时间范围、指标、维度

Observation 1: 获取原始数据

Thought 2: 数据处理
  - 排序、计算同比、找出异常

Action 2: 调用可视化Skill
  - 生成柱状图或折线图

Final Answer: 自然语言回答 + 图表
```

**前端功能要点**：
- 对话式交互界面
- 数据可视化（ECharts/D3.js）
- 查询历史记录
- 图表导出功能

---

**选题2：智能写作**

**业务场景**：
- 生成月度运行分析报告
- 撰写故障处理总结
- 自动生成会议纪要框架
- 公文格式自动排版

**ReAct设计要点**：
```
Thought 1: 理解写作需求
  - 文档类型：报告/总结/纪要
  - 内容范围：哪些数据、哪些事件
  - 格式要求：正式程度、结构

Action 1: 检索模板Skill
  - 根据文档类型获取对应模板

Observation 1: 获取模板结构

Thought 2: 内容填充规划
  - 每个章节需要什么数据
  - 哪些需要查询，哪些可以生成

Action 2: 批量查询数据Skill
  - 并行查询多个数据源

Observation 2: 获取所有数据

Thought 3: 内容生成
  - 按模板结构组织内容
  - 润色语言风格

Action 3: 生成文档

Final Answer: 完整文档 + 格式排版
```

**前端功能要点**：
- 富文本编辑器（支持Markdown）
- 模板选择面板
- 实时预览
- 导出功能（Word/PDF）

---

**选题3：工单处理**

**业务场景**：
- 自动识别工单类型（故障报修、业务咨询、投诉建议）
- 根据内容分派给对应部门
- 提供处理建议和历史案例参考
- 预测处理时长

**ReAct设计要点**：
```
Thought 1: 工单解析
  - 关键词提取
  - 意图识别
  - 紧急程度判断

Action 1: 分类推理
  - 使用分类模型或规则引擎

Observation 1: 分类结果（如：故障报修-配电线路）

Thought 2: 查询知识库
  - 查找同类工单历史
  - 查找处理流程和规范

Action 2: 检索知识Skill
  - 查询工单知识库

Observation 2: 获取相关案例和流程

Thought 3: 生成处理建议
  - 参考历史案例
  - 结合当前情况
  - 给出分派建议

Action 3: 生成建议报告

Final Answer: 
  - 工单分类
  - 建议分派部门
  - 处理建议
  - 参考案例
```

**前端功能要点**：
- 工单列表（支持筛选、排序）
- 工单详情页
- AI分析面板（分类、建议、相似案例）
- 处理流程跟踪

| 术语 | 英文 | 解释 |
|------|------|------|
| Agent | Agent | 智能体，能自主决策和执行任务的AI系统 |
| ReAct | ReAct | 推理+行动模式，让AI显式思考再行动 |
| MCP | Model Context Protocol | 模型上下文协议，标准化AI与环境的交互 |
| Skill | Skill | 技能，封装好的可复用能力单元 |
| LLM | Large Language Model | 大语言模型，如GPT-4、文心一言等 |
| RAG | Retrieval-Augmented Generation | 检索增强生成，结合知识库的生成技术 |
