# 项目设计：工单处理

## 项目概述

**项目名称**：电力工单智能处理系统

**项目目标**：自动分析工单内容，智能分配处理部门，生成处理建议，提升工单处理效率

**目标用户**：电力公司客服人员、工单处理人员、管理人员

**复杂度**：⭐⭐⭐⭐

---

## 一、业务场景

### 典型需求

```
场景1：故障报修
用户来电："我们家停电了，地址是XX小区XX栋XX号"
→ 自动识别：故障报修、地址提取、派发给运维班组

场景2：业务咨询
用户咨询："我想申请新装电表，需要什么材料？"
→ 自动识别：业务咨询、回复标准答案、或转人工

场景3：投诉建议
用户投诉："上个月电费突然变高很多，我要投诉"
→ 自动识别：投诉类型、生成处理方案、升级优先级

场景4：信息查询
用户查询："我想查一下上个月的用电量"
→ 自动识别：信息查询、调用系统接口、返回数据
```

### 用户痛点

- 传统方式：人工分类慢、分配不准确、处理建议依赖经验
- 智能工单：自动分类 → 智能分配 → 生成建议 → 快速处理

### Agent流程设计（ReAct + 循环模式）

```
用户提交工单
     ↓
【Thought】分析工单类型
"用户反馈停电问题，可能是故障报修类型"
     ↓
【Action】调用分类识别
分类模型：故障报修、业务咨询、投诉建议、信息查询、其他
     ↓
【Observation】分类结果
{
  "type": "故障报修",
  "confidence": 0.95,
  "sub_type": "停电"
}
     ↓
【Thought】提取关键信息
"需要提取地址、联系方式、故障描述等关键信息"
     ↓
【Action】调用信息提取
提取字段：地址、联系电话、故障描述、时间、紧急程度
     ↓
【Observation】信息提取成功
{
  "address": "XX小区XX栋XX号",
  "phone": "138****1234",
  "description": "家里突然停电",
  "time": "2024-03-03 14:30",
  "urgency": "高"
}
     ↓
【Thought】查询处理部门
"故障报修需要分配给运维班组，查询对应区域的责任班组"
     ↓
【Action】调用HTTP接口
查询该地址对应的责任班组
     ↓
【Observation】找到责任班组
{
  "department": "城东运维班",
  "contact": "李工 139****5678",
  "estimated_time": "30分钟"
}
     ↓
【Thought】生成处理建议
"根据工单类型和历史案例，生成处理建议"
     ↓
【Action】查询知识库并生成建议
召回相似工单处理方案
     ↓
【Final Answer】返回完整处理结果
{
  "工单编号": "GD20240303001",
  "分类": "故障报修-停电",
  "关键信息": {...},
  "处理部门": "城东运维班",
  "处理建议": "1. 核实停电范围 2. 安排现场勘查 3. 预计30分钟内响应",
  "紧急程度": "高",
  "相关案例": [相似工单列表]
}
```

**案例文档示例**：
```markdown
# 停电故障处理规范

## 适用场景
用户反馈家中停电，无预兆，非计划停电

## 标准处理流程

### 1. 信息核实（5分钟内）
- 确认停电范围（单户/整栋/整个小区）
- 询问是否有预兆（跳闸/线路老化/天气原因）
- 记录联系方式和详细地址

### 2. 初步判断
- 单户停电 → 可能是内部线路问题或电表故障
- 整栋停电 → 可能是单元配电箱故障
- 整个小区 → 可能是变压器或主线故障

### 3. 派单处理
- 紧急程度：高
- 响应时间：30分钟内到达现场
- 责任部门：对应区域运维班

### 4. 现场处理
- 携带必要工具和备件
- 排查故障原因
- 现场修复或临时处理
- 告知客户恢复情况

### 5. 回访确认
- 处理后24小时内回访
- 确认问题是否彻底解决
- 记录客户满意度

## 注意事项
- 优先保障老人、医院等重要客户
- 恶劣天气需注意安全
- 复杂故障需上报调度中心
```
